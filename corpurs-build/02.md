--------------------------------------------------
GOAL
--------------------------------------------------

Create a production-shaped RAG v1 that can answer questions like:
- what are all rules about level
- what invalidates a breakout setup
- what is the difference between acceptance and false breakout
- show all examples tied to retest
- what lessons discuss volume confirmation
- what rule conflicts exist across lessons
- what rules depend on higher timeframe context

The system must be provenance-first and deterministic where possible.

--------------------------------------------------
NON-GOALS
--------------------------------------------------

Do NOT:
- build prediction models
- build feature store
- build label generation
- rewrite Step 2 schema
- make markdown the primary retrieval source
- rely on one giant LLM answer call
- hide citations / provenance
- build a polished frontend UI

A tiny explorer or JSON endpoints are fine.
Primary deliverable is the retrieval system.

--------------------------------------------------
TECHNICAL DIRECTION
--------------------------------------------------

Implement a hybrid RAG stack with these layers:

1. Relational metadata store
   Recommended: Postgres

2. Vector retrieval
   Recommended: pgvector if using Postgres
   Alternative only if project already has strong reason: Qdrant

3. Lexical retrieval
   Implement BM25-style or equivalent lexical search over retrieval documents
   If BM25 infra is too heavy, implement a clean lexical scorer abstraction with a first practical backend

4. Object storage abstraction
   For now, local filesystem paths are acceptable for screenshots / lesson assets
   Design an adapter so S3-style object storage can be added later

5. Retrieval API
   FastAPI recommended

6. Evaluation harness
   Deterministic evaluation script + representative query set

Important:
Choose one primary stack and implement it fully.
Do not create multiple half-finished backends.

Recommended concrete choice for this repo:
- Postgres + pgvector
- FastAPI
- local filesystem asset resolver
- lexical index built from persisted retrieval docs

--------------------------------------------------
RETRIEVAL UNITS
--------------------------------------------------

Index these units separately.

A. Rule cards
Best for rule lookup and explicit teacher guidance

B. Knowledge events
Best for nuanced conditions, invalidations, exceptions, comparisons, warnings

C. Evidence refs
Best for timestamped examples, visual grounding, human inspection

D. Concept nodes / concept relations
Best for concept expansion, aliasing, related-rule discovery, query rewrite

Each unit must remain distinguishable in storage and retrieval.
Do NOT flatten everything into one text blob.
