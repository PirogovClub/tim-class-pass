Below is the **implementation brief for a junior coding agent** for the **next step: Stage 5.5 — proposal-assisted review**.

This step sits on top of the already accepted adjudication storage, APIs, workstation UI, and tiering foundation. The project rule is still the same: **AI proposes, human adjudicates bounded decisions, and proposals must never silently mutate curated state**. Stage 5.5 must add durable proposal records, scoring, queue generation, and light UI visibility without turning into autonomous curation.

# Stage 5.5 — Junior Agent Implementation Pack

## 1. Mission

Implement the **first production-ready proposal layer** for adjudication.

The goal is to help the reviewer find the best next work faster. This is not a training pipeline, not a dashboard project, and not an auto-merge engine. The system should create durable proposal records, turn them into useful review queues, and show the reviewer **why** the item was proposed. It must not change curated state unless the reviewer submits a normal adjudication decision. 

## 2. Scope for this branch

Implement **only the first, highest-value slice**:

### In scope

1. proposal schema and durable persistence
2. proposal generator for:

   * `duplicate_candidate`
   * `merge_candidate`
   * `canonical_family_candidate`
3. proposal-backed queues
4. proposal visibility in existing UI
5. proposal linkage to later human decisions
6. carry-forward 5.4 cleanup tests:

   * orphan-row purge test
   * stronger recompute idempotence test

### Explicitly out of scope for this branch

Do **not** implement yet:

* evidence support proposals
* concept-link proposals
* related-rule relation proposals
* ambiguity/conflict proposals
* auto-approval policies
* export
* metrics dashboard
* model-serving or background worker architecture

These can come later. Right now, keep the branch narrow and operational. The Stage 5 design requires proposal generation, persistence, and queueing, but it also says to keep the workflow bounded and auditable.  

---

## 3. Non-negotiable rules

The agent must follow these rules:

1. **No hidden mutation**

   * Proposal generation must never directly update reviewed state, canonical family membership, or tier state.
   * Curated state still changes only through explicit adjudication decisions. 

2. **Durable proposals**

   * Proposals are first-class stored records, not temporary in-memory suggestions. 

3. **Bounded reviewer task**

   * The reviewer should see local evidence and candidate context.
   * The system must not require reviewer memory across the whole corpus. 

4. **Minimal UI**

   * Extend the accepted workstation. Do not redesign it.
   * Proposal context should be visible, but clearly non-authoritative. 

5. **Deterministic first**

   * For this branch, prefer deterministic proposal generation over LLM-generated free text.
   * The project already has deterministic matching helpers such as `normalize_text_for_match()` and `simple_text_similarity()` in `pipeline/component2/rule_reducer.py`; reuse them instead of inventing a new opaque scoring system.

## 8. Storage schema to add

In `storage.py` and repository init, add a table similar to this:

```sql
CREATE TABLE IF NOT EXISTS adjudication_proposal (
    proposal_id TEXT PRIMARY KEY,
    proposal_type TEXT NOT NULL,
    source_target_type TEXT NOT NULL,
    source_target_id TEXT NOT NULL,
    related_target_type TEXT,
    related_target_id TEXT,
    proposal_status TEXT NOT NULL,
    score REAL NOT NULL,
    score_breakdown_json TEXT,
    rationale_summary TEXT NOT NULL,
    signals_json TEXT,
    evidence_refs_json TEXT,
    adjudication_snapshot_json TEXT NOT NULL,
    tier_snapshot_json TEXT NOT NULL,
    queue_priority REAL NOT NULL,
    dedupe_key TEXT NOT NULL UNIQUE,
    generator_version TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    last_generated_at TEXT NOT NULL,
    accepted_decision_id TEXT,
    stale_reason TEXT
);
```

Add indexes:

* `(proposal_type, proposal_status)`
* `(source_target_type, source_target_id)`
* `(related_target_type, related_target_id)`
* `(queue_priority DESC)`
* `(generator_version)`

### Dedupe rule

A proposal must be unique by logical candidate, not by run.

Use a `dedupe_key` such as:

* duplicate: `duplicate|rule_card|<low_id>|rule_card|<high_id>`
* merge: `merge|rule_card|<low_id>|rule_card|<high_id>`
* canonical family candidate: `canonical|rule_card|<rule_id>|canonical_rule_family|<family_id>`

Sort pair IDs so the same pair does not produce two rows.

## 10. Proposal generation logic

## 10.1 General rule

For this branch, implement **deterministic heuristic proposal generation**, not LLM reasoning.

Reason:

* easier to audit
* easier to test
* stable
* enough for first useful duplicate/merge/canonical candidate surfacing

## 10.2 Data inputs

Use:

* authoritative target inventory from `CorpusTargetIndex`
* reviewed state from repository
* tier state from materialized tier rows
* canonical family membership/state from repository
* rule card content/provenance from explorer/read model
* text similarity utilities already present in `pipeline/component2/rule_reducer.py` such as `normalize_text_for_match()` and `simple_text_similarity()`

## 10.3 First slice: only rule-card proposals

Generate proposals only for `rule_card` sources in this branch.

### Eligible source rules

A rule can be considered for candidate generation only if:

* it exists in authoritative inventory
* it is not rejected
* it is not unsupported
* it is not already marked duplicate of another rule
* it is not in unresolved state due to broken linkage
* it has enough basic structure to compare:

  * non-empty rule text
  * concept or subconcept available

### Candidate pairing strategy

Do **not** compare every rule against every other rule if the corpus is large.

Use cheap pruning first:

* same concept, or
* same subconcept, or
* normalized text token overlap above a small threshold, or
* shared evidence refs / source events, or
* same canonical family candidate neighborhood from existing bundle logic

### Duplicate candidate scoring

Use a clear deterministic formula, for example:

* normalized text similarity: `0.45`
* same concept: `0.15`
* same subconcept: `0.10`
* shared evidence ref overlap: `0.10`
* shared source event overlap: `0.10`
* both non-Gold / unresolved improvement opportunity: `0.05`
* both from same lesson penalty: `-0.05` if this creates trivial self-neighbors only

Clamp result to `[0.0, 1.0]`.

Threshold:

* `>= 0.88` => create `duplicate_candidate`

### Merge candidate scoring

These are similar but not identical rules that may belong in one canonical family.

Suggested scoring:

* text similarity: `0.35`
* same concept: `0.20`
* same subconcept: `0.10`
* condition/invalidation compatibility: `0.10`
* shared evidence overlap: `0.10`
* one rule already in active family and other not: `+0.10`
* both have clear provenance: `+0.05`

Threshold:

* `>= 0.72 and < 0.88` => create `merge_candidate`

### Canonical family candidate scoring

This is for a rule not yet clearly attached to a family.

For each eligible rule:

* compare against active canonical families
* use family canonical title + normalized summary + member rule texts
* pick top family only if above threshold

Scoring:

* family-summary text similarity: `0.40`
* concept match: `0.20`
* subconcept match: `0.10`
* overlap with member rules: `0.20`
* family active bonus: `0.10`

Threshold:

* `>= 0.75` => create `canonical_family_candidate`

### Very important

Do not produce all three proposal types for the same source pair blindly.

Suggested precedence:

1. duplicate candidate
2. merge candidate
3. canonical family candidate

If a pair qualifies as duplicate, do not also emit merge for that same pair.

## 11. Rationale generation

Do not call an LLM for rationale in this branch.

Build short deterministic rationale text from the strongest signals.

Examples:

* `"High text overlap, same concept, shared evidence refs"`
* `"Moderate semantic overlap, same subconcept, one rule already belongs to active family fam_012"`
* `"Rule matches active family summary and overlaps with 3 existing family members"`

Also store `signals_json`, for example:

```json
{
  "text_similarity": 0.91,
  "concept_match": true,
  "subconcept_match": true,
  "shared_evidence_count": 2,
  "shared_source_event_count": 1,
  "family_id": "fam_012"
}
```

This keeps the proposals inspectable and auditable.

## 13. Staleness rules

A proposal becomes stale when:

* the source target disappears from authoritative inventory
* the related target disappears
* the source or related target gets a later adjudication decision that makes the proposal no longer relevant
* the generator reruns and the candidate no longer meets threshold
* the source rule becomes duplicate/merged/rejected/unsupported in a way that resolves the issue

Implement a staleness pass after each generation run:

* proposal rows seen again => refresh `updated_at`, `last_generated_at`, score, rationale, snapshots
* proposal rows not regenerated => mark `proposal_status = stale`, not hard delete

Do not delete history unless there is a clear maintenance reason.
