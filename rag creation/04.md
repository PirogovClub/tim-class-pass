  * `rule_count = 10`
  * `len(top_rules) = 10`
  * `event_count = 10`

That is exactly the same failure mode as before.

### Why this matters

A concept detail page is supposed to tell the analyst how much material exists for that concept. Right now it only tells them how many preview cards were included.

### Exact fix

In `get_concept_detail()` compute full totals separately:

```python
rule_docs = [doc for doc in docs if doc.get("unit_type") == "rule_card"]
event_docs = [doc for doc in docs if doc.get("unit_type") == "knowledge_event"]
evidence_count = sum(1 for doc in docs if doc.get("unit_type") == "evidence_ref")

top_rules = rule_docs[:10]
top_events = event_docs[:10]
```

Then pass explicit totals into `build_concept_detail()`:

```python
return build_concept_detail(
    concept_id=meta.get("concept_id") or concept_id,
    aliases=self._repo.get_concept_aliases(concept_id),
    neighbors=self._repo.get_concept_neighbors(concept_id),
    top_rules=top_rules,
    top_events=top_events,
    lessons=lessons,
    evidence_count=evidence_count,
    rule_count=len(rule_docs),
    event_count=len(event_docs),
)
```

And update `build_concept_detail()` to accept `rule_count` and `event_count` rather than deriving them from the preview arrays.

### 2) `/browser/facets` is still truncated

In `pipeline/explorer/service.py`:

* line 269: `top_k=100`
* line 273: `return self.search(req).facets`

And inside `search()`:

* line 193: `facets = self._compute_facets(docs)`

So facets are still computed on the already-truncated page docs, not the full filtered result set.

The saved sample proves this again:

* `explorer_build_manifest.json` says `total_docs_loaded = 1591`
* `browser_api_samples/facets.json` shows:

  * `by_unit_type = {"rule_card": 100}`

That is not a real corpus facet view. It is just the first 100 browse docs.

### Why this matters

This makes the facets endpoint misleading:

* counts are artificially capped
* unit type distribution is biased by browse sort order
* evidence/events can disappear from facets even when they exist in the filtered corpus

### Exact fix

Do not compute facets through paginated search.

Add a dedicated internal path, for example:

```python
def _docs_for_facets(self, query: str, filters: BrowserSearchFilters) -> list[dict[str, Any]]:
    if not query.strip():
        return [
            doc
            for doc in self._sort_browse_docs(self._repo.get_all_docs())
            if self._matches_filters(doc, filters)
        ]

    result = self._retriever.search(
        query=query,
        top_k=1000,  # or configurable exact facet limit
        unit_types=filters.unit_types or None,
        lesson_ids=filters.lesson_ids or None,
        min_confidence=filters.min_confidence_score,
    )
    return [
        hit["resolved_doc"]
        for hit in result["top_hits"]
        if self._matches_filters(hit["resolved_doc"], filters)
    ]
```

Then:

```python
def get_facets(...):
    docs = self._docs_for_facets(query or "", filters or BrowserSearchFilters())
    return self._compute_facets(docs)
```

If you intentionally want “facets of current page only,” then the endpoint should be renamed or documented that way. But that would not match the Step 4.1 expectation we were using for acceptance.
