# Step 4 — detailed overview

## What Step 4 is

Step 4 is **not another retrieval-quality phase** and it is **not ML yet**.

Step 4 is the first **knowledge product layer** on top of your accepted hybrid RAG.

Its job is to turn the retrieval system into a **usable analyst workspace** where you can:

* search by concept or natural-language phrasing
* inspect rules in a clean structured way
* inspect supporting evidence and timestamps
* inspect lesson context
* traverse related concepts and rules
* compare related rules across lessons
* prepare clean inputs for later label design and ML task definition

So Step 4 is basically:

> **Rule browser / analyst interface built on top of Step 3 RAG + corpus outputs**

---

## What Step 4 is not

Do **not** let Step 4 drift into these things yet:

* no prediction model
* no label generator
* no backtesting
* no trade execution logic
* no autonomous “strategy synthesis”
* no editing / write-back workflow
* no large UI app with auth, user accounts, dashboards, etc.

Keep it **read-only, grounded, structured, and deterministic**.

---

## Why Step 4 matters

Right now you have:

* structured corpus
* accepted retrieval system
* accepted retrieval quality hardening
* good eval + regression coverage

But you still do not yet have a **human-friendly analysis surface**.

Without Step 4, the system is still mostly a technical retrieval engine.

Step 4 gives you the first place where you can actually do knowledge work like:

* “show me all rules about false breakout entry”
* “compare stop-loss rules across lessons”
* “which rules are transcript-primary only?”
* “what evidence supports this rule?”
* “what other concepts connect to this one?”
* “what lessons disagree or differ in phrasing?”

That is the bridge between:

* extraction / retrieval
  and
* later labeling / modeling / research

---

# Step 4 goals

## Primary goal

Create a **read-only analyst browser** that exposes your knowledge base in a deterministic, inspectable format.

## Secondary goals

Make it easier to:

* audit retrieval quality manually
* inspect provenance fast
* inspect evidence fast
* compare rules across lessons
* define future label ontology and ML tasks

## Success criterion

A human should be able to answer these questions **without reading raw JSON files**:

1. What rules exist for a concept?
2. Which lessons teach them?
3. What evidence supports them?
4. Are they transcript-primary, mixed, or visually supported?
5. What related rules / concepts should I inspect next?
6. Which rules are good candidates for future label design?

### Mapping rules

For `rule_card` docs:

* `title` comes from `title`
* `snippet` should prefer `rule_text_ru`, then `rule_text`, then `short_text`
* `evidence_count = len(evidence_ids)`
* `support_basis`, `teaching_mode`, `evidence_requirement`, `confidence_score` should always be surfaced if present

For `knowledge_event` docs:

* `snippet` should prefer the normalized event statement text
* title can be a compact form like:

  * concept / event type
  * or lesson + event type if concept missing

For `evidence_ref` docs:

* title should show something human-meaningful, not just ID
* snippet should prefer visual summary or evidence text
* timestamps must always be exposed

### Important

Do not just dump raw docs.
The browser needs **clean cards** and **clean details**.

## D. Browser search behavior

This matters a lot.

### If `query` is non-empty

Use the existing accepted retriever:

* call `HybridRetriever.search(...)`
* transform hits into browser result cards
* enrich cards with support/evidence fields
* optionally group into:

  * rules
  * events
  * evidence
  * concepts

### If `query` is empty

This becomes **browse mode**.

In browse mode:

* use repository filtering only
* sort deterministically

Recommended sort order for browse mode:

1. `rule_card`
2. `knowledge_event`
3. `evidence_ref`
4. `concept_node`
5. `concept_relation`

Within a unit type:

* higher confidence first
* then lesson_id
* then doc_id

This lets the browser open even without a query.

---

## E. Add browser-specific filtering

Current Step 3 filters are too narrow for a real analyst browser.

Add filters for:

* `support_basis`
* `evidence_requirement`
* `teaching_mode`

You can implement them in the browser service first, without modifying the core retriever contract.

### Suggested strategy

1. retrieve candidates
2. post-filter in browser service
3. recompute returned hit count
4. compute facets on filtered results

Later, if needed, you can push some of this deeper into store/retriever.

That keeps Step 4.1 safe.
