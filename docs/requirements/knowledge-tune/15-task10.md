---

## Output artifact

* `concept_graph.json`

and wire it into the standard artifact export path.

---

## Graph design

Keep the graph lesson-local for now.

### Node types

Use a small controlled set:

* `concept`
* `subconcept`
* `condition`
* `invalidation`
* `exception`
* `pattern`

Do not introduce deeper ontology yet.

### Edge types

Use a small controlled set:

* `has_subconcept`
* `related_to`
* `has_condition`
* `has_invalidation`
* `has_exception`
* `co_occurs_with`

That is enough for now.

---

## Suggested schema

Use a deterministic JSON shape like:

```python
{
  "lesson_id": "...",
  "graph_version": "1.0",
  "nodes": [
    {
      "node_id": "concept:level",
      "label": "level",
      "node_type": "concept",
      "lesson_ids": ["..."],
      "source_rule_ids": ["rule_001", "rule_017"],
      "aliases": []
    }
  ],
  "edges": [
    {
      "edge_id": "edge_001",
      "source": "concept:level",
      "target": "subconcept:false_breakout",
      "relation": "has_subconcept",
      "weight": 3,
      "source_rule_ids": ["rule_017", "rule_021"]
    }
  ],
  "stats": {
    "node_count": 0,
    "edge_count": 0
  }
}
```

You may adapt field names to existing schema conventions, but preserve this logical structure.

---

## Concept extraction rules

### Create concept nodes from:

* `RuleCard.concept`
* `RuleCard.subconcept`

Normalize values by:

* lowercasing
* trimming whitespace
* replacing internal repeated spaces
* optionally slugifying for IDs

Example:

* label: `False Breakout`
* normalized node_id: `subconcept:false_breakout`

### Create optional secondary nodes from rule structure

Create nodes for conditions / invalidations / exceptions only when the text is short, non-empty, and reusable.

Good:

* `"entry only after confirmation candle closes above level"`
* `"invalid if breakout has no volume support"`

Bad:

* long prose paragraph
* vague teaching commentary
* placeholder text

Use conservative thresholds.

---

## Edge construction rules

### 1. Concept → subconcept

If a rule has both `concept` and `subconcept`:

* create `concept -> subconcept` with `has_subconcept`

### 2. Concept/subconcept → condition

For each short normalized condition:

* attach to the closest owning node using `has_condition`

Prefer subconcept if present, otherwise concept.

### 3. Concept/subconcept → invalidation

For each short normalized invalidation:

* attach using `has_invalidation`

### 4. Concept/subconcept → exception

For each short normalized exception:

* attach using `has_exception`

### 5. Co-occurrence edges

If two concept/subconcept nodes repeatedly appear in the same rule cards:

* create `related_to` or `co_occurs_with`

Only create this edge if co-occurrence count exceeds a small threshold, for example:

* threshold `>= 2`

This prevents noisy one-off links.

---

## Weighting rules

Use deterministic integer weights.

Suggested logic:

* `has_subconcept`: weight = number of supporting rules
* `has_condition`: weight = number of rules carrying that condition
* `has_invalidation`: weight = number of rules carrying that invalidation
* `has_exception`: weight = number of rules carrying that exception
* `co_occurs_with`: weight = co-occurrence count

---

## Deduplication rules

You must deduplicate aggressively but conservatively.

Two nodes should merge if normalized labels match exactly.

Do **not** use fuzzy matching in this step.

Two edges should merge if:

* same `source`
* same `target`
* same `relation`

Then combine:

* `weight`
* `source_rule_ids`
