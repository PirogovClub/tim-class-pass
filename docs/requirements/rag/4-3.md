## 6.2 New backend capabilities

### A. Rule comparison

Add a browser-facing comparison response for multiple rules.

Recommended endpoint:

```text
POST /browser/compare/rules
```

Request shape:

* `rule_ids: list[str]`
* optional `include_related_context: bool = true`

Response should include, for each rule:

* `rule_id`
* title
* lesson id
* concept / subconcept
* rule text
* conditions
* invalidation
* exceptions
* support basis
* evidence requirement
* confidence
* timestamps summary
* linked evidence count
* linked source event count
* related rule count

And a **comparison summary** section:

* `shared_concepts`
* `shared_lessons`
* `shared_support_basis`
* `differences`
* `possible_relationships`

Do not try to make the backend infer deep semantic truth. Keep the comparison summary deterministic and field-based.

### B. Lesson comparison

Add:

```text
POST /browser/compare/lessons
```

Request:

* `lesson_ids: list[str]`

Response should include lesson-by-lesson summaries:

* counts by unit type
* counts by support basis
* top concepts
* top rules
* top evidence
* concept overlap matrix or simple overlap section
* unique concepts per lesson
* shared concepts
* approximate shared rule families if available

### C. Related-rules workflow

Add:

```text
GET /browser/rule/{doc_id}/related
```

This should return:

* same concept rules
* same rule-family rules
* source-lesson sibling rules
* maybe event-linked variants if available

For each related item, return:

* result card
* relation reason, e.g.:

  * `same_concept`
  * `same_family`
  * `same_lesson`
  * `linked_by_evidence`
  * `cross_lesson_overlap`

### D. Concept traversal helper

You already have concept detail and neighbors. Extend it with explicit linked-entity traversal support.

Either:

* extend `GET /browser/concept/{concept_id}` with richer linked sections, or
* add:

```text
GET /browser/concept/{concept_id}/rules
GET /browser/concept/{concept_id}/lessons
```

I recommend adding both.

These should return browser-friendly result cards, not raw docs.

## 7.2 Required UI workflows

### A. Compare rules workflow

User can:

* select 2–4 rules
* open compare view
* inspect side-by-side differences

The rule compare page should show:

* one column per rule
* sticky comparison header
* shared vs different concept info
* conditions/invalidation/exceptions aligned by section
* support badges
* timestamps summary
* evidence/source counts
* buttons/links to open full rule detail

Also include a small “comparison summary” card at the top:

* shared concepts
* same lesson vs cross-lesson
* same support basis or not
* likely variant / duplicate / sibling / contrastive relationship

### B. Compare lessons workflow

User can:

* select 2–4 lessons
* open compare view
* inspect overlap and differences

The lesson compare page should show:

* lesson summary cards
* counts by unit type
* counts by support basis
* shared concepts
* unique concepts per lesson
* top rules per lesson
* quick links into lesson detail / concept pages

### C. Related-rules workflow

From any rule detail page, user can:

* open related rules
* see grouped reasons:

  * same concept
  * same family
  * same lesson
  * cross-lesson linked
* jump to any related rule detail

This should feel like a real analyst “next things to inspect” workflow.

### D. Concept traversal workflow

From concept detail page, user can:

* open all rules for concept
* open all lessons for concept
* move through neighbors
* use the concept as a stable traversal anchor

This is Step 4.3 concept navigation.
