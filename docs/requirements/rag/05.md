# Step 4.2 UI Architecture Brief for the Agent

## 1. Objective

Build a **browser/explorer UI** for analysts that makes the accepted Step 4.1 backend usable.

The UI must let a user:

* search or browse explorer entities
* inspect rule details
* inspect evidence details
* inspect concept details and neighbors
* inspect lesson summaries
* filter results with facets
* deep-link and share current state via URL

This is the first UI layer over the accepted explorer backend. It is intentionally **thin**. It is not a large app, not an admin system, not a charting system, and not an ML dashboard.  

## 2. Non-goals

Do **not** implement:

* auth
* annotations / write-back
* saved workspaces
* graph visualization
* export workflows
* rule comparison workflows
* lesson comparison workflows
* ML predictions
* market-data charts
* anything that bypasses the explorer API

Those belong to later steps or separate tracks.  

---

## 3. Architectural principles

1. **UI only consumes `/browser/*`**
   Never read raw corpus or retrieval files from the frontend. The explorer read-model layer already exists specifically to prevent that. 

2. **Read-only by design**
   The entire UI is a browser for knowledge inspection.

3. **URL is the source of truth for navigation state**
   Query, filters, grouping, and selected entity pages should all be reconstructable from the URL.

4. **Server state and client state stay separate**
   Use TanStack Query for API data. Use local/UI state only for transient view concerns.

5. **Step 4.3 and 4.4 extensibility must be preserved**
   Do not hard-code the UI in a way that blocks later compare views, traversal workflows, and export bundles.

## 5. Route map

Implement these frontend routes:

```text
/                          -> redirect to /search
/search                    -> search + browse page
/rule/:docId               -> rule detail
/evidence/:docId           -> evidence detail
/concept/:conceptId        -> concept detail
/lesson/:lessonId          -> lesson detail
```

Use query params for search state:

```text
/search?q=...
&group=unit_type
&lesson=...
&concept=...
&unit_type=...
&support_basis=...
&evidence_requirement=...
&teaching_mode=...
&min_confidence_score=...
```

Optional:

* `mode=search|browse`
* `page` only if you add client-side pagination later
* `sort` only if you intentionally expose UI sort overrides

Do not add compare/export routes yet. Reserve those namespaces for Step 4.3 and 4.4:

* `/compare/...`
* `/export/...`

## 6. API integration contract

The UI must consume only the explorer endpoints already defined and accepted:

* `GET /browser/health`
* `POST /browser/search`
* `GET /browser/rule/{doc_id}`
* `GET /browser/evidence/{doc_id}`
* `GET /browser/concept/{concept_id}`
* `GET /browser/concept/{concept_id}/neighbors`
* `GET /browser/lesson/{lesson_id}`
* `GET /browser/facets`  

### API client rules

Create a small typed client in `src/lib/api/browser.ts`.

Requirements:

* one function per endpoint
* central base URL configuration via `VITE_BROWSER_API_BASE`
* parse responses through Zod before exposing them to components
* normalize errors into one UI-friendly shape
* no component should call `fetch` directly

Example client surface:

```ts
searchBrowser(request: BrowserSearchRequest): Promise<BrowserSearchResponse>
getRuleDetail(docId: string): Promise<RuleDetailResponse>
getEvidenceDetail(docId: string): Promise<EvidenceDetailResponse>
getConceptDetail(conceptId: string): Promise<ConceptDetailResponse>
getConceptNeighbors(conceptId: string): Promise<ConceptNeighborListResponse>
getLessonDetail(lessonId: string): Promise<LessonDetailResponse>
getFacets(query: string, filters: BrowserSearchFilters): Promise<FacetResponse>
healthCheck(): Promise<HealthResponse>
```

## 7. Data model handling

Do not hand-wave the contracts in the UI.

Create:

* `schemas.ts` with Zod parsers for all explorer responses
* `types.ts` inferred from those Zod schemas

This protects the UI from backend drift and makes failures explicit.

### Minimum schemas to define

* `BrowserSearchFilters`
* `BrowserSearchRequest`
* `ResultCard`
* `BrowserSearchResponse`
* `RuleDetailResponse`
* `EvidenceDetailResponse`
* `ConceptDetailResponse`
* `ConceptNeighborListResponse`
* `LessonDetailResponse`
* `FacetResponse`
* `ApiErrorResponse`

If the backend already has documented example payloads in `browser_api_samples/`, use those as fixtures for schema validation and frontend tests.

## 8. State management

### Server state

Use TanStack Query for:

* search results
* facets
* detail pages
* health check

### URL state

Use router search params for:

* `q`
* filters
* grouping mode

### Local UI state

Use React local state only for:

* open/closed filter panels
* dialogs
* selected tab inside a detail page
* minor layout preferences

Do not introduce Redux or another global state layer.

## 9. UI screens and required behavior

## 9.1 Search page

This is the main page.

### Must support

* free-text query
* empty-query browse mode
* filters panel
* active filter chips
* grouped or flat result display
* loading, empty, and error states
* click-through into detail views

### Search mode

If `q` is non-empty:

* call `POST /browser/search`
* show result cards
* render support/provenance badges clearly

### Browse mode

If `q` is empty:

* still call the same search UI flow, but the backend will behave as browse mode
* clearly label that the user is browsing, not searching

### Cards

Each result card should show:

* title
* unit type
* snippet
* lesson id or display label
* confidence if present
* support basis
* evidence requirement
* timestamps when available
* click target

### Grouping

Support:

* `group=none`
* `group=unit_type`

This should be URL-driven.

---

## 9.2 Filters panel

Expose the explorer filters that Step 4.1 already defined:

* `lesson_ids`
* `concept_ids`
* `unit_types`
* `support_basis`
* `evidence_requirement`
* `teaching_mode`
* `min_confidence_score` 

Requirements:

* reusable filter chips at the top of the page
* “clear all” action
* each facet section collapsible
* counts visible beside facet values
* preserve state in URL
* fetch `/browser/facets` based on current query + current filters

Important:

* keep page-level search results and global facet results conceptually separate in the UI if needed
* do not mislabel page-scoped data as corpus-global data

## 9.3 Rule detail page

Must render:

* title
* normalized rule text
* conditions
* invalidations
* exceptions
* comparisons if present
* support/provenance badges
* timestamps
* linked evidence
* source events
* related rules

This page should feel like the most polished one because it is central to the analyst workflow. Step 4 exists so a human can inspect rules and evidence clearly.

## 9.4 Evidence detail page

Must render:

* evidence title/snippet
* lesson id
* timestamps or time ranges
* support basis
* linked rules
* linked events
* any role or strength metadata already exposed by the backend

Keep this page text-first and evidence/provenance-first. Do not invent chart rendering here.

## 9.5 Concept detail page

Must render:

* concept id / name
* aliases
* neighbors
* lesson coverage
* top rules
* top events
* counts

The neighbors panel must be clickable and lead to related concept pages or filtered search flows.

No graph visualization yet. Step 4.1 explicitly limited concept traversal to a lightweight neighbor surface. 

---

## 9.6 Lesson detail page

Must render:

* counts by unit type
* counts by support basis
* top concepts
* top rules
* top evidence 

This page should help answer “what does this lesson contribute to the corpus?”

## 10. Layout and UX guidance

Use a two-panel desktop layout:

* left: filters and navigation context
* right: results or detail view

On smaller screens:

* filters collapse into a drawer
* detail pages remain single-column

### Shared UI elements

Build reusable:

* `SupportBadges`
* `EntityHeader`
* `LinkedEntityList`
* `TimestampList`
* `CountPills`
* `FilterChips`

### Styling guidance

Use Tailwind + shadcn/ui.
Prefer:

* cards
* badges
* accordions
* tabs only where they genuinely improve clarity
* drawers/sheets for mobile filters

Avoid:

* heavy animation
* charts
* glassmorphism
* unnecessary dashboard chrome

This UI is for inspection speed, not visual theatrics.

---

## 11. Error handling

Implement one normalized error model.

Cases to support:

* 400 bad request
* 404 missing entity
* wrong unit type response
* network error
* schema validation failure in frontend parsing

Behavior:

* page-level errors show a clear error panel
* entity pages show “Not found” when 404
* invalid response shape should fail loudly in development, gracefully in production

## 12. Performance and caching

### Query keys

Use stable TanStack Query keys, for example:

* `['browser-search', params]`
* `['browser-facets', params]`
* `['rule-detail', docId]`
* `['evidence-detail', docId]`
* `['concept-detail', conceptId]`
* `['concept-neighbors', conceptId]`
* `['lesson-detail', lessonId]`

### Caching policy

Suggested initial defaults:

* search results: short stale time
* detail pages: moderate stale time
* facets: short stale time
* health: long stale time

Do not over-optimize. Correctness and clarity matter more right now.

6. open rule detail from search result
7. open evidence detail from linked evidence
8. open concept detail and click neighbor
9. open lesson detail from a result card
10. invalid entity route shows not-found

These preserve the accepted Step 3.1 behaviors in the UI layer, not just the backend regression suite.
