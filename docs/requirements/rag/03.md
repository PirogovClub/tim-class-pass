## Blocking issue 1: concept detail counts are wrong

In `ExplorerService.get_concept_detail()`, the code slices:

* `top_rules = ...[:10]`
* `top_events = ...[:10]`

Then `build_concept_detail()` sets:

* `rule_count = len(top_rules)`
* `event_count = len(top_events)`

That means `rule_count` and `event_count` are **not actual counts**. They are just **the number of returned preview items**, capped at 10.

### Why this matters

The Step 4.1 concept detail is supposed to tell the analyst **concept coverage**, not just preview list length. Right now, any concept with more than 10 rules or events will be understated.

### What to change

In `service.py`, compute totals from the full `docs` set first, then pass the truncated preview lists separately.

Use logic like this:

```python
rule_docs = [doc for doc in docs if doc.get("unit_type") == "rule_card"]
event_docs = [doc for doc in docs if doc.get("unit_type") == "knowledge_event"]
evidence_count = sum(1 for doc in docs if doc.get("unit_type") == "evidence_ref")

top_rules = rule_docs[:10]
top_events = event_docs[:10]
```

Then either:

* extend `build_concept_detail()` to accept `rule_count` and `event_count`, or
* build the response directly in `service.py`.

Example change:

```python
return build_concept_detail(
    concept_id=meta.get("concept_id") or concept_id,
    aliases=self._repo.get_concept_aliases(concept_id),
    neighbors=self._repo.get_concept_neighbors(concept_id),
    top_rules=top_rules,
    top_events=top_events,
    lessons=lessons,
    evidence_count=evidence_count,
    rule_count=len(rule_docs),
    event_count=len(event_docs),
)
```

and in `views.py`:

```python
def build_concept_detail(
    concept_id: str,
    *,
    aliases: list[str],
    neighbors: list[dict[str, Any]],
    top_rules: list[dict[str, Any]],
    top_events: list[dict[str, Any]],
    lessons: list[str],
    evidence_count: int = 0,
    rule_count: int = 0,
    event_count: int = 0,
) -> ConceptDetailResponse:
    return ConceptDetailResponse(
        concept_id=concept_id,
        aliases=list(aliases),
        top_rules=[build_result_card(candidate) for candidate in top_rules],
        top_events=[build_result_card(candidate) for candidate in top_events],
        lessons=list(lessons),
        neighbors=[ConceptNeighbor(**neighbor) for neighbor in neighbors],
        rule_count=rule_count,
        event_count=event_count,
        evidence_count=evidence_count,
    )
```

### How to verify

Add a fixture where one concept has:

* 12 rule cards
* 15 events
* 3 evidence refs

Then assert:

* `len(top_rules) == 10`
* `len(top_events) == 10`
* `rule_count == 12`
* `event_count == 15`
* `evidence_count == 3`

## Blocking issue 2: `/browser/facets` is truncated and misleading

This is the bigger product issue.

`ExplorerService.get_facets()` creates a `BrowserSearchRequest` with `top_k=100`, then calls `self.search(req)`, and `search()` computes facets on the already-truncated `docs` list.

So `/browser/facets` is **not** returning facets for the full filtered explorer result set. It is returning facets for **the first 100 returned docs**.

The saved sample strongly supports this problem: the bundle manifest says `total_docs_loaded = 1591`, while the saved `facets.json` shows only `{"rule_card": 100}` under `by_unit_type`, which is exactly what you would expect from the current `top_k=100` implementation and browse ordering. 

### Why this matters

This makes the facets endpoint unreliable for the UI or analyst surface:

* counts can be far too low
* entire unit types can disappear from facets
* the first page ordering biases the facet distribution

For example, in empty-query browse mode the first 100 are rule-first, so the facet payload can falsely imply the result set is all rules.

### What to change

Do **not** compute facets through paginated search.

Create a separate internal method that:

* gathers the full filtered doc set for browse mode, or
* gathers the full filtered hit set for query mode, up to an explicit exact-count limit if needed.

At minimum:

```python
def _filtered_docs_for_facets(
    self,
    query: str,
    filters: BrowserSearchFilters,
) -> list[dict[str, Any]]:
    if not query.strip():
        return [
            doc
            for doc in self._sort_browse_docs(self._repo.get_all_docs())
            if self._matches_filters(doc, filters)
        ]

    result = self._retriever.search(
        query=query,
        top_k=1000,  # or configurable exact-facet candidate limit
        unit_types=filters.unit_types or None,
        lesson_ids=filters.lesson_ids or None,
        min_confidence=filters.min_confidence_score,
    )
    return [
        hit["resolved_doc"]
        for hit in result["top_hits"]
        if self._matches_filters(hit["resolved_doc"], filters)
    ]
```

Then `get_facets()` should call `_compute_facets()` directly on that full filtered set, not on `self.search(req).facets`.

### How to verify

For empty-query `/browser/facets`:

* sum of `by_unit_type.values()` should equal the number of filtered docs, not 100
* multiple unit types should appear when the corpus contains them

For filtered cases:

* add a test where the true filtered set is >100 docs
* assert the facet total exceeds 100 when appropriate

## Medium issue 3: test coverage is below the stated Step 4.1 requirement

The Step 4.1 requirement set explicitly called for API tests for:

* `/browser/search`
* `/browser/rule/{id}`
* `/browser/evidence/{id}`
* `/browser/concept/{id}`
* `/browser/concept/{id}/neighbors`

It also explicitly called for regression coverage for:

* stop-loss example evidence-first

But the submitted explorer API tests only cover:

* health/search
* rule detail
* wrong type for rule

* timeframe regression

What is missing from the explorer API test layer:

* evidence detail endpoint
* concept detail endpoint
* neighbors endpoint
* lesson detail endpoint
* facets endpoint
* missing-id / 404 cases across all detail endpoints
* daily-level actionable-first regression

### Why this matters

This is exactly why the two bugs above slipped through. The tests prove the surface exists, but they do not sufficiently prove the surface is **correct**.

### What to change

Add explicit API tests for all required endpoints and one service/view test each for:

* concept detail total counts vs preview counts
* facets full filtered counts vs truncated page counts

Also add the missing regression test:
