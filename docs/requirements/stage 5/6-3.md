## 3. Core invariants you must preserve

### 3.1 Structured corpus remains the source of truth

The RAG system must ingest the structured knowledge artifacts and the 6.2 unified corpus outputs, not scrape markdown as primary input. The project’s core knowledge artifacts are the structured JSON outputs like `knowledge_events`, `rule_cards`, `evidence_index`, and `concept_graph`, while markdown is a derived projection.  

### 3.2 Provenance is mandatory

Every retrieved answer and retrieval hit must preserve traceability:

* source lesson
* rule/event/evidence ids
* timestamps where applicable
* evidence linkage
* concept linkage where applicable

The project has repeatedly emphasized that normalized rules and evidence must remain traceable to source lesson content.  

### 3.3 Hybrid means hybrid

Do not build “vector search only” and call it done. The roadmap explicitly says:

* lexical/BM25
* vector similarity
* concept/graph expansion
* reranking

must be part of the design. 

### 3.4 Retrieval units must stay separate

Do not flatten everything into one giant text blob. The roadmap explicitly says to index at least these units separately:

* rule cards
* knowledge events
* evidence refs
* concept nodes / graph relations 

### 3.5 Do not drift into model training

This step is the RAG database and retrieval API. The project is explicit that the prediction layer is separate and later, and should not be trained on lesson prose.

## 8. Retrieval unit design

You must index these units separately.

### 8.1 Rule cards

Best for high-level rule lookup:

* “What does the teacher say about false breakout after retest?”
* “Show rules about level acceptance above” 

Store at least:

* `rule_id`
* lesson id
* concept/subconcept
* rule title/text
* conditions
* invalidations
* evidence refs
* source event ids
* confidence / tier where available
* embedding text

### 8.2 Knowledge events

Best for nuance and fine-grained retrieval:

* conditions
* invalidations
* exceptions
* comparisons 

Store at least:

* `event_id`
* lesson id
* event type
* normalized text
* concept/subconcept
* timestamps
* evidence refs
* confidence
* embedding text

### 8.3 Evidence refs

Best for timestamped examples, grounding, and human inspection. 

Store at least:

* `evidence_id`
* lesson id
* linked rule ids
* source event ids
* timestamps
* frame/screenshot refs
* compact summary
* example role
* embedding text

### 8.4 Concept nodes / graph relations

Best for concept expansion, query rewrite, and related-rule discovery. 

Store at least:

* concept node ids
* canonical names
* aliases
* parent/related relations
* lesson support counts
* optional synthetic searchable text for retrieval

### Important

Do not merge these into one “document” type.
Each retrieval unit should remain typed, because later reranking and answer formatting depend on that separation.

## 11. Hybrid retrieval requirements

This is the core of the step.

### 11.1 Retrieval modes you must implement

At minimum:

* lexical retrieval
* vector retrieval
* concept/alias expansion
* reranking

The roadmap explicitly names all four. 

### 11.2 Recommended lexical approach

Use BM25 or equivalent lexical scoring over the normalized retrieval text.

### 11.3 Concept expansion

Implement query expansion using:

* canonical concept ids/names
* aliases
* normalized variants
* graph neighbors where appropriate

The roadmap explicitly says query flow should expand through canonical IDs and aliases. 

### 11.4 Reranking

Rerank combined candidates using a transparent scoring policy that can include:

* concept match strength
* lexical score
* vector score
* provenance/evidence availability
* confidence/tier
* exact phrase hits
* retrieval unit type preference by query intent

### 11.5 First-slice priority

For first implementation, prioritize a strong deterministic hybrid scorer over an opaque “LLM rerank everything” approach.

## 12. Query flow requirements

Implement the query flow explicitly.

1. detect main concept from query
2. expand through canonical IDs and aliases
3. retrieve rule cards + events
4. rerank using provenance / confidence / concept match
5. return structured answer with timestamps and evidence 

### Required behavior

The retrieval API should not just return a flat list of text chunks.
It should return structured results with:

* matched unit type
* id
* score components if possible
* lesson id
* concept/subconcept
* timestamps if applicable
* evidence refs
* linked related items where possible
