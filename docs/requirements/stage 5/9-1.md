SEQUENCE JUSTIFICATION RULES
================================

Sequence should only be recommended if evidence suggests problems that are plausibly due to missing temporal dynamics, such as:

- class failures that depend on multi-bar order/shape rather than static summary features
- performance instability on patterns where sequence evolution matters
- symbolic and tabular models both missing patterns that humans describe temporally
- feature importance / failure analysis suggesting summary features are insufficient

If recommending sequence next, the system must also define a narrow Step 10 brief including:
- candidate architecture family:
  - TCN
  - LSTM
  - transformer on OHLCV sequences
- input sequence schema
- sequence length
- timeframes
- target labels
- evaluation plan
- calibration plan
- explanation limitations

Do not recommend sequence just because it sounds more advanced.

================================
VISION JUSTIFICATION RULES
================================

Vision should only be recommended if evidence suggests that chart-image structure likely adds signal BEYOND structured price data.

This is a very high bar.

Vision should NOT be recommended by default.

Vision may only be recommended if:
- there is a concrete representation gap that sequence/tabular are unlikely to cover well
- there is evidence that chart layout/pattern geometry is central
- the decision report explains why sequence is not the preferred next step
- the step10 brief defines:
  - chart rendering contract
  - image standardization
  - label alignment policy
  - evaluation plan
  - proof requirement that vision beats structured-price alternatives

If the evidence does not clearly support this, the correct answer is not vision.

================================
UPSTREAM-FIRST RULES
================================

The decider must support the conclusion that advanced models are NOT the next bottleneck.

This should happen when evidence suggests the bigger problems are:
- label ambiguity
- low class support
- weak calibration stability
- fragile fold performance
- poor regime coverage
- near-constant / low-signal features
- unreliable policy-level usefulness
- insufficient dataset size for meaningful complex models

This is not a failure.
This is often the correct decision.

FAILURE-MODE ANALYSIS REQUIREMENTS
================================

You must analyze at least:

1. CLASS-LEVEL WEAKNESS

2. FOLD / TIME STABILITY
- whether performance is unstable across forward folds

3. CALIBRATION QUALITY
- whether probabilities are materially unreliable or drift across folds

4. POLICY-LEVEL UTILITY
- whether thresholded policies are actually useful
- whether combined symbolic+model policy improves selection quality

5. DATA / SUPPORT LIMITS
- whether some classes or regimes have too little support for complex models

6. FEATURE / REPRESENTATION LIMITS
- whether failures suggest missing temporal or shape information rather than just noisy data
