# Junior Coding Agent Prompt — Stage 6.1 Freeze the Knowledge System as v1

## READ THIS FIRST — DEFINITION OF DONE

**This task is NOT done when coding is finished.**
**This task is done only when ALL of the following are true:**

1. The schema/contracts are frozen and versioned.
2. A stable lesson-level export contract is documented.
3. A `lesson_registry.json` is produced for processed lessons.
4. A corpus-level validator exists and runs successfully.
5. The validator checks required artifacts and contract shape.
6. Provenance requirements are enforced or explicitly failed.
7. The tiny `summary_ru` schema tidy-up is finalized if still pending.
8. Tests are written and actually run.
9. Example outputs are generated.
10. Audit handoff files are prepared.
11. The final audit **zip file** is created.

If any one of these is missing, the task is **not complete**.

The project roadmap explicitly says **Phase A** is to freeze the knowledge system as v1 before building the DB, corpus store, or hybrid RAG, and it names the exact deliverables above. It also says you should **not build retrieval or ML infrastructure on a moving schema**.

## 3. Core invariants you must preserve

### 3.1 JSON outputs remain the source of truth

The structured knowledge outputs are the system’s core artifacts:

* `knowledge_events.json`
* `rule_cards.json`
* `evidence_index.json`
* `concept_graph.json`

Markdown is derived, not source of truth. Do not invert that. 

### 3.2 Provenance is mandatory

Every normalized rule must preserve provenance:

* source lesson
* timestamps
* source event ids
* linked evidence refs

The project is explicit that no normalized rule should become detached from original lesson evidence.

### 3.3 Visuals are evidence, not first-class prose payload

Do not let schema freeze accidentally promote verbose visual narration into main RAG content. Visuals should remain supporting evidence with timestamp/frame/screenshot linkage and compact summaries.

### 3.4 Stable contracts before infrastructure

This step exists because later retrieval and ML work must not depend on a moving schema. Do not treat this as a casual cleanup. It is the contract-lock step for Phase A.

## 7. Required contract rules

### 7.1 Artifact presence rules

A processed lesson must have:

* `knowledge_events.json`
* `rule_cards.json`
* `evidence_index.json`
* `concept_graph.json`

If any required primary artifact is missing, validation must fail. The project’s earlier acceptance guidance already treated these as the essential outputs.

### 7.2 Required provenance rules

At minimum, `rule_cards.json` entries must preserve:

* `lesson_id`
* `source_event_ids`
* `evidence_refs` or explicit empty list

This was already established earlier as an important acceptance rule and must now become part of the frozen contract.

### 7.3 Derived-output rule

If `rag_ready.md` or `review_markdown.md` are present, document that they are derived from normalized structured outputs, not raw dense text. This behavior was already explicitly required earlier and should be preserved in the contract narrative.

### 7.4 ID stability rule

Document expectations for stable IDs at the lesson level:

* `lesson_id`
* `event_id`
* `rule_id`
* `evidence_id`
* concept IDs if present

You are not building corpus-wide global IDs yet, but you should freeze the lesson-level contract so later corpus-wide IDs can build on it cleanly. The next phase will require stable global IDs at corpus level. 

### 7.5 Version stamping rule

Each artifact or registry entry should point to:

* relevant schema version
* source artifact version if available
* contract version

This is the entire point of schema freeze.

## 10. Corpus validator requirements

Implement a validator that supports at least:

### 10.1 Structural checks

* required files exist
* JSON parses
* top-level shape is valid
* required fields exist
* required field types are valid

### 10.2 Provenance checks

* every rule card has `lesson_id`
* every rule card has `source_event_ids`
* every rule card has `evidence_refs` or explicit empty list
* evidence refs resolve if possible
* timestamps are present where required by the current schema

### 10.3 Cross-artifact checks

Where possible, validate:

* `rule_cards.source_event_ids` refer to events in `knowledge_events.json`
* `evidence_refs` refer to entries in `evidence_index.json`
* concept references match `concept_graph.json` where applicable

### 10.4 Registry consistency checks

* registry entries point to real files
* registry counts match actual file contents
* registry schema version fields are present
* validation status in registry matches actual validation result

### 10.5 Contract version checks

* lesson artifacts and registry entries map to known schema versions
* unknown schema versions fail validation or warn explicitly depending on strict mode

### 10.6 Modes

If practical, support:

* strict mode
* lenient/report-only mode

But strict mode is the default expectation for schema freeze.
