# Instruction for Junior Coding Agent — Stage 5.2

## Mission

Implement **Stage 5.2: Review APIs and Queue APIs** on top of the **accepted Stage 5.1 adjudication foundation**.

At the end of this work, the backend must be able to:

* fetch review items through stable APIs
* fetch reviewed state and review history
* fetch canonical family details and family members
* fetch a full review bundle for a target
* submit adjudication decisions through API
* return updated reviewed state after a decision
* expose deterministic unresolved / by-target / next-item queue APIs

This step is **backend API only**.

Do **not** build UI.
Do **not** build AI proposal generation.
Do **not** build Gold/Silver/Bronze logic.
Do **not** build export.
Do **not** build auth platform.
Do **not** do a broad refactor of unrelated explorer code.

---

# What success means

Stage 5.2 is successful only if all of the following are true:

1. Review items can be fetched through stable APIs.
2. Reviewed state and review history are exposed cleanly.
3. A review bundle endpoint exists and returns enough bounded context for later UI.
4. Decision submission works through API with full validation.
5. Queue APIs exist for unresolved and by-target workflows.
6. Queue ordering is deterministic.
7. Family detail and family members can be fetched safely.
8. Errors are machine-readable and consistent.
9. Tests cover both positive and negative paths.
10. Stage 5.3 UI can start without guessing backend behavior.

# Important design principles

## 1) Hide storage internals

The API must not expose:

* raw SQLite rows
* repository-specific details
* resolver internals
* internal DB-only field names unless they are part of the public contract

The UI must talk to **API contracts**, not storage tables.

## 2) One review page should be mostly one fetch

The future UI should not need 7 fragile requests just to review one item.

Create a **review bundle** endpoint that returns:

* target summary
* reviewed state
* history
* family info if relevant
* small linked context if available

## 3) Write APIs must be explicit and safe

Decision submission must:

* validate request payload
* validate target/decision compatibility
* validate reviewer existence
* validate family existence where needed
* append decision
* refresh state
* return updated state

## 4) Queue APIs should be simple first

No advanced prioritization yet.

Stage 5.2 queue logic should be:

* deterministic
* understandable
* documented
* easy to test

## 5) No hidden mutation

No route should change adjudicated state unless it is an explicit decision-submission endpoint.

## Patch 9 — Build queue service

### Goal

Create deterministic queue logic using Stage 5.1 resolved-state data.

### Tasks

Implement queue functions like:

```python
list_unresolved_queue(...) -> QueueListResponse
list_queue_by_target(target_type: ReviewTargetType, ...) -> QueueListResponse
get_next_queue_item(queue_name: str, target_type: ReviewTargetType | None = None) -> QueueItemResponse | None
```

### Required first-version queue policy

Keep it simple and deterministic.

Recommended unresolved criteria:

* current status indicates `needs_review`
* or `is_ambiguous = True`
* or `is_deferred = True`
* or no meaningful final review state exists

Use the actual Stage 5.1 state model.
If some fields differ, document the exact implemented rule.

### Ordering

Use stable ordering such as:

1. unresolved/priority bucket
2. oldest `last_reviewed_at` first
3. `target_id` ascending as tie-breaker

If `last_reviewed_at` is null, define and document fallback behavior.

### Important

Do not add AI ranking.
Do not add proposal queues.
Do not infer “best review value” yet.

### Deliverable

* `pipeline/adjudication/queue_service.py`

### Tests

Add tests for:

* unresolved queue returns only unresolved items
* by-target queue filters correctly
* next-item returns the first deterministic item
* empty queue returns a stable empty result
* ordering is reproducible

### Success criteria

* queue logic is stable and testable
* results are deterministic
* no scoring complexity is introduced
