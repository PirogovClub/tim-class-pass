# Stage 5.3 — Minimal Review Workstation UI

## Agent Context Document

## Purpose of Stage 5.3

Stage 5.3 builds the **minimal adjudication workstation UI** on top of the accepted Stage 5.1 storage layer and Stage 5.2 API layer.

By the end of Stage 5.3, the system should no longer only be able to:

* store adjudication decisions
* expose adjudication APIs
* expose queues and review bundles

It should also be able to:

* show a review queue to a human reviewer
* open a reviewable item
* show current reviewed state and decision history
* show bounded review context from the review bundle
* let the reviewer submit decisions
* let the reviewer move efficiently to the next item
* support minimal duplicate/merge review through a compare workflow

Stage 5.3 is the first point where the adjudication system becomes **operational for real human review**.

---

# Why Stage 5.3 exists

Stage 5.1 created the adjudication foundation:

* typed review targets
* typed decision history
* durable storage
* canonical family structure
* materialized reviewed state

Stage 5.2 created the adjudication backend interface:

* read APIs
* review bundle API
* decision submission API
* family APIs
* queue APIs

But without Stage 5.3, a reviewer still cannot work efficiently at scale.

Stage 5.3 must solve the real workflow problem:

* a reviewer cannot hold the full corpus in memory
* a reviewer needs a guided, bounded context
* a reviewer needs a queue, a review page, decision controls, and next-item flow
* a reviewer needs enough context to adjudicate without reconstructing everything manually

So Stage 5.3 is the first usable **human review surface**.

---

# Stage 5.3 Goals

## Primary goals

* build a usable review queue UI
* build a usable review item UI
* surface review bundle context clearly
* let the reviewer submit decisions through the UI
* support next-item workflow
* support minimal duplicate/merge compare workflow
* keep the UI thin and grounded in Stage 5.2 APIs

## Non-goals

Stage 5.3 should **not** become:

* polished final product UI
* analytics dashboard
* graph visualization tool
* AI proposal generation interface
* batch editing platform
* multi-user collaboration suite
* auth/permissions system
* complex keyboard-shortcut power tool set
* export or ML tooling

This stage is a **minimal workstation**, not a platform.

---

# Global Design Principles

## 1) The UI must reduce reviewer effort

The UI exists to make adjudication easier, not to expose every internal detail.

## 2) The UI must not assume reviewer memory

The page should show enough local context that the reviewer can decide without remembering the whole corpus.

## 3) One item review should feel self-contained

A reviewer should be able to:

* open an item
* understand its state
* inspect its context
* make a decision
* move on

without constant navigation across many screens.

## 4) The UI must rely on Stage 5.2 APIs

No direct storage access.
No UI-side reconstruction of database logic.
No hidden rules in frontend-only code.

## 5) Decisions must map clearly to adjudication types

The UI should not blur or rename core decisions so much that they stop matching the adjudication model.

## 6) Compare should be minimal but real

Duplicate and merge review need a compare surface, but it does not need to be a large generic compare product yet.

---

# What Stage 5.3 should include

At minimum, Stage 5.3 should include:

## 1) Review queue page

A queue page where the reviewer can:

* see unresolved items
* filter by target type
* understand why an item is in queue
* open an item into review
* get to next item efficiently

## 2) Review item page

A review page where the reviewer can see:

* target summary
* current reviewed state
* latest decision
* queue reason
* decision history
* family info if relevant
* bounded review context from review bundle

## 3) Decision action panel

A decision panel where the reviewer can:

* choose a decision
* optionally enter note/reason
* provide related target id when required
* submit the decision
* see updated state

## 4) History panel

A visible history panel showing:

* prior decisions
* reviewer ids
* timestamps
* notes
* order of decisions

## 5) Family panel

A panel showing:

* canonical family info
* family members
* member roles
* linkage of current item to family if any

## 6) Minimal compare workflow

A reviewer must be able to compare two items for:

* duplicate decisions
* merge decisions
* canonical-family decisions

This can be:

* a separate compare page
* a drawer
* a modal
* or a split panel

Any of those is acceptable if it works.

## 7) Next-item flow

After making a decision, the reviewer should be able to:

* stay on current item
* return to queue
* go to next queue item

without friction.
