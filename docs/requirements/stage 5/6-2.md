## 3. Core invariants you must preserve

### 3.1 Structured JSON remains the source of truth

The project’s core lesson artifacts are the structured JSON set:

* `knowledge_events.json`
* `rule_cards.json`
* `evidence_index.json`
* `concept_graph.json`

Markdown is derived, not primary. The corpus builder must merge the structured artifacts, not scrape markdown.

### 3.2 Provenance is mandatory

Every corpus-level rule/event/evidence item must retain traceability back to its lesson-level origin. The project has been explicit that normalized rules must preserve:

* source lesson id
* source event ids
* evidence refs
* timestamps where possible

### 3.3 Corpus IDs must be stable

This step is where stable corpus-level IDs are introduced. The roadmap explicitly requires stable global keys for artifacts such as `lesson_id`, `event_id`, `rule_id`, `evidence_id`, and `node_id`.

### 3.4 No hidden mutation of lesson artifacts

The corpus builder reads lesson artifacts and emits corpus outputs. It must not silently rewrite accepted lesson artifacts in place.

### 3.5 Keep RAG concerns out of this step

Do not drift into retrieval implementation here. This step is the corpus layer that the later hybrid RAG system will ingest. The hybrid RAG design comes in the next phase.

## 8. Stable ID policy

You must define and document a stable ID policy.

### 8.1 Goal

The same input lesson corpus, with the same artifact contents, must produce the same corpus IDs.

### 8.2 Requirement

Do **not** use random UUIDs.

Use deterministic IDs derived from stable inputs such as:

* lesson slug / lesson id
* lesson-local artifact id
* normalized concept or relation fields where required
* explicit version-aware hashing if needed

### 8.3 Recommended approach

Examples:

* `lesson_id` = stable existing lesson id from registry
* `event_id` = preserve existing stable event id if lesson-level IDs are already stable; otherwise derive deterministic corpus id from `(lesson_id, local_event_id)`
* `rule_id` = preserve existing stable rule id if possible; otherwise derive deterministic corpus id from `(lesson_id, local_rule_id)`
* `evidence_id` = same principle
* `node_id` = canonical concept-based deterministic id, not arbitrary per-run numbering

### 8.4 Important

If lesson-level IDs are already stable and contract-frozen, prefer **stable preservation + namespace-safe lifting** over inventing entirely new ids.

This makes downstream debugging much easier.

## 9. Enrichment requirements

You must implement the required corpus enrichments.

### 9.1 Canonical concept frequencies

Compute how often each canonical concept appears across lessons and corpus items.

At minimum include:

* concept name
* lesson count
* rule count
* event count

### 9.2 Rule families

Build corpus-level grouping for similar or related rules.

Important:

* do not reinvent Stage 5 adjudication family structure if exported reviewed family structure already exists and can be reused
* but if Phase B is operating mainly over lesson artifacts, you may need a first corpus-family grouping layer based on canonical concept + normalized rule attributes

Be explicit in docs about what “rule families” means in this step.

### 9.3 Concept-to-rule mappings

Build a reverse index from concept/subconcept to linked rule ids.

### 9.4 Cross-lesson concept overlap

Measure which concepts appear across multiple lessons and where they overlap.

### 9.5 Concept alias registry

Build a registry that can hold:

* canonical concept id/name
* aliases
* multilingual variants
* normalized forms

The knowledge-layer architecture explicitly calls for canonical IDs / aliases / multilingual fields in the knowledge base.

## 11. Validation requirements

Implement a corpus validator for this step.

### 11.1 Structural checks

Validate that:

* required corpus files exist
* JSON / JSONL parses
* required fields exist
* IDs are unique where they must be unique

### 11.2 Cross-reference checks

Validate that:

* corpus rule `source_event_ids` resolve to corpus events
* corpus rule `evidence_refs` resolve to corpus evidence
* corpus evidence `linked_rule_ids` resolve to corpus rules
* concept links referenced by corpus items resolve to corpus concept graph when applicable

### 11.3 Provenance checks

Validate that:

* every corpus rule retains source `lesson_id`
* every corpus rule retains source events and evidence refs fields
* timestamps are present where the underlying artifact required them

### 11.4 Registry replay checks

Validate that:

* every lesson in registry that is marked valid participates correctly in corpus ingestion
* missing or invalid lessons are reported cleanly
* corpus counts can be tied back to per-lesson counts

### 11.5 Determinism checks

Validate that:

* repeated corpus builds from the same input produce the same IDs and same normalized outputs, ignoring explicitly documented volatile fields
