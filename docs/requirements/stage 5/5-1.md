# Important design principles

## 1) Append-only decision history

The review history should be the **source of truth** for adjudication actions.

This means:

* every review action becomes a persisted decision record
* do not overwrite history destructively
* current state should be derived from decisions, not replace them

## 2) Resolved state should exist separately

Do **not** make the rest of the system replay raw decision history every time.

You need:

* append-only decision log
* materialized or computed current state layer

## 3) Canonical structure must be explicit

Do not leave “merged” meaning in notes only.

You need first-class representation for:

* canonical family
* member rule relationships
* family status

## 4) Typed decisions only

No free-form-only review logic.

For example:

* `approve`
* `reject`
* `duplicate_of`
* `merge_into`
* `concept_valid`
* `concept_invalid`
* `evidence_strong`
* `evidence_partial`
* `relation_valid`
* `relation_invalid`
* `defer`
* `ambiguous`

These must be enums or strict typed values.

## 5) Keep provenance

Each decision must know:

* target type
* target id
* who/what made the decision
* timestamp
* reason/notes
* artifact/source version if available

## Patch 5 — Add resolver/current-state layer

### Goal

Convert raw decision history into current reviewed state.

This is the most important logic patch.

### Tasks

Implement a resolver module that can:

* load all decisions for a target
* apply them in order
* compute current resolved state
* persist/update the resolved-state table

### Recommended approach

Create explicit resolver functions per target type.

For example:

```python
resolve_rule_card_state(target_id: str, decisions: list[ReviewDecision]) -> RuleCardReviewedState
resolve_evidence_link_state(...)
resolve_concept_link_state(...)
resolve_related_rule_relation_state(...)
```

### Resolver behavior examples

#### Rule card

If decisions are:

1. `needs_review`
2. `ambiguous`
3. `duplicate_of` -> rule_123

Then the resolved state should reflect:

* latest decision = `duplicate_of`
* `is_duplicate = True`
* `duplicate_of_rule_id = "rule_123"`
* `is_ambiguous` should remain only if your policy keeps historical flags active
  or be turned into current derived status only from latest decision

Pick one policy and document it.

#### Evidence link

If latest decision is `evidence_partial`, current support state = partial.

#### Concept link

If latest decision is `concept_invalid`, current link status = invalid.

#### Related-rule relation

If latest decision is `relation_valid`, relation status = valid.

### Important design choice

Be explicit whether current state is:

* last-decision-wins
  or
* rule-based aggregation across history

For junior implementation, I recommend:

## Use **last-decision-wins** for the first version,

with a few explicit derived flags:

* duplicate
* deferred
* unsupported
* canonical_family_id

This is simpler, safer, and enough for Stage 5.1.

### Deliverable

* `pipeline/adjudication/resolver.py`

### Tests

Add tests covering:

* no decisions
* one decision
* multiple decisions same target
* duplicate decision with related target
* merge decision into family
* deferred after approve
* approve after ambiguous
* restart/reload recomputation

### Success criteria

* current state can be deterministically reconstructed from history

## Patch 6 — Persist resolved state

### Goal

Do not only resolve in memory. Persist the current resolved state into state tables.

### Tasks

After appending a decision:

* recompute the target’s resolved state
* upsert the corresponding state table row

### Rules

* do not mutate decision history
* do not store only raw state without history
* keep state rows deterministic

### Recommended pattern

Repository/service function:

```python
append_decision_and_refresh_state(...)
```

This should:

1. append decision
2. load target history
3. resolve current state
4. write current state row

### Deliverable

* repository/service update
* state upsert logic

### Tests

Add tests that:

* append a decision
* verify resolved state table is updated
* append a later decision
* verify state changes correctly
* verify history still contains all previous decisions

# Business rules to implement now

These are enough for Stage 5.1.

## Rule 1

Decision history is append-only.

## Rule 2

Resolved state is derived from decision history.

## Rule 3

Last decision wins for current status, unless you need one of these explicit derived flags:

* duplicate
* deferred
* unsupported
* canonical_family_id

## Rule 4

Canonical family membership is stored separately and explicitly.

## Rule 5

A decision that depends on another target must store that relation explicitly via `related_target_id` or equivalent typed linkage.

## Rule 6

Notes are helpful, but never replace typed decisions.
